# PLAN.md — Dropping LiteLLM & Migrating to a Clean LangChain‑Only Implementation

## Context

The Multi‑Agent Dashboard currently uses a dual‑path architecture:

- **LiteLLM path** (`USE_LITELLM=true`): Unified provider abstraction via `langchain‑litellm`, with dynamic capability detection, tool‑calling adaptation, and JSON‑Schema workarounds.
- **Non‑LiteLLM path** (`USE_LITELLM=false`): Direct provider‑specific integrations (OpenAI, DeepSeek, Ollama) with structured output, file attachments (text for OpenAI), and web search (OpenAI) support, but limited tool handling and multimodal file processing.

LiteLLM’s reliance on hardcoded model‑specification files (`model_prices_and_context_window.json`, `provider_endpoints_support.json`) introduces unpredictable detection and opaque behavior. As the MVP only needs to support **OpenAI, DeepSeek, and Ollama**, we can remove the LiteLLM layer entirely, migrate its enhancements back to a unified provider‑specific LangChain implementation, and gain full control over the three providers.

**Reference:** Full strategy documented in [`docs/implementation‑strategies/drop_litellm_migration.md`](docs/implementation-strategies/drop_litellm_migration.md).

## Goal

**Remove all LiteLLM dependencies** while preserving every feature currently available in the LiteLLM path:

- Structured output (JSON Schema, function‑calling, json‑mode)
- Tool calling (native web‑search, DuckDuckGo function tool)
- Multimodal file handling (images, PDFs, text files)
- User‑controlled endpoint overrides and API‑version preferences (`use_responses_api`)
- UI capability toggles (`provider_features`, tool checkboxes, structured‑output toggle)
- Cost/latency instrumentation and strict‑schema validation

**Outcome:** A simpler, more predictable codebase with zero LiteLLM code, fully functional for the three locked‑in providers, and a clear path to add new providers via static capability mapping.

## Execution Guidelines

**Critical instructions for the executing agent:**

1. **Case‑by‑case evaluation** – Do **not** rely solely on this plan. For each step, examine the actual old and new code paths, understand their behavior, and verify changes are correct for the specific context.

2. **Stop for ambiguity** – If any requirement, behavior, or code path is ambiguous or unclear, **stop work immediately** and wait for user approval/decision. Do not guess or make assumptions.

3. **Fetch latest documentation** – Before implementing provider‑specific logic:
   - Use `agentic_fetch` to search for latest LangChain provider library documentation
   - Check provider API specifications (OpenAI, DeepSeek, Ollama)
   - Verify model‑specific capabilities (vision, tool‑calling, structured output)
   - Confirm parameter names and supported features

4. **Test before deletion** – Always test new implementations thoroughly before removing LiteLLM code. Maintain dual‑path architecture (`USE_LITELLM=false`) until validated.

5. **Preserve working code** – When extending existing implementations to new providers, preserve working logic for original providers. Add new branches/adapters rather than rewriting.

6. **Prefer modularization over nested branches** – Split code into separate helper functions, modules, and packages. Avoid deep nesting of provider‑specific branches; instead create focused modules (e.g., `tool_integration/provider_tool_adapter.py`, `multimodal_handler.py`). Move existing code affected by the refactor into dedicated modules where appropriate.

7. **Dynamic capability detection, no prompt‑based tool emulation** – Any capability detection must be dynamic and supported by the provider API (no heuristics or hardcoded model names). Never implement prompt‑based tool‑calling emulation; tool invocation via prompts remains the user's responsibility via manual agent prompt configuration.

Refer to the detailed strategy in [`docs/implementation‑strategies/drop_litellm_migration.md`](docs/implementation-strategies/drop_litellm_migration.md) for complete guidance.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Provider‑specific bugs in LangChain integrations. | Keep existing provider‑specific branches; add unit tests for edge cases. |
| Capability mapping becomes outdated as models evolve. | Treat mapping as advisory only; rely on user configuration (`provider_features`) as primary source of truth. |
| Heuristic‑based capability detection leading to incorrect behavior. | Use provider APIs for dynamic capability detection when needed; never rely on hardcoded model names or heuristics. |
| Regression in tool‑handling for existing agents. | Thoroughly test each provider with the new adapter before deleting LiteLLM path. |
| File‑handling breakage for multimodal use cases. | Keep fallback to text‑only concatenation when provider lacks image support. |
| Performance impact from removing LiteLLM caching. | LangChain’s `ChatModelFactory` already caches model instances; no loss. |
